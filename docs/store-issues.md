# 商店问题分析

本文只基于当前地图工程仓库内可见代码做判断。由于 `smallcard_store` 库源码不在当前仓库，所以部分结论是“高可信推断”，不是最终定论。

## 一、商店链路总览

当前商店打开流程可以概括为：

1. `ui/script/gui/page/MainPage/template.lua` 里的 `MainPage` 绑定点击事件 `单击商店按钮`
2. `ui/script/gui/ui_response/单击商店按钮.lua` 转到 `validator_34975774`
3. `ui/script/trigger_validator.ts` 创建 `客户端点击商店` 事件对象
4. `script/trigger_module_main_1.ts` 监听 `客户端点击商店`
5. 服务端调用 `smallcard_store.打开商店(..., "$$p_3b4f.loot_pool.游戏商店.root")`

## 二、问题 1：商店关闭之后无法再打开

### 现象理解

你当前描述的是：

- 商店最初能看到或能打开
- 关闭以后，再次点击商店入口无法重新打开

### 已确认的代码事实

#### 1. 主页面里商店面板默认就是显示的

`ui/script/gui/page/MainPage/template.lua` 里可见：

- `MainPage` 自身 `show = true`
- `smallcard_store.商店_商店面板` 也是 `show = true`
- 商店关闭按钮和多个商店子控件也默认 `show = true`

这说明首次看到商店界面，不一定是“点击后才创建”，也可能是页面加载时就已挂载。

#### 2. 再次打开依赖服务端触发器

服务端真正执行打开商店的是：

- `script/trigger_module_main_1.ts`
- 函数 `_TRIG_接收客户端打开商店_FUNC`
- 调用 `smallcard_store.打开商店(base.player(1).get_hero(), base.player(1).get_hero(), "$$p_3b4f.loot_pool.游戏商店.root")`

也就是说，“重开商店”是否成功，关键不在模板文件，而在点击事件有没有重新走到服务端。

#### 3. 客户端点击逻辑目前只看到“构造事件对象”

`ui/script/trigger_validator.ts` 中：

- `validator_34975774` 只创建了 `new p_3b4f.客户端点击商店(...)`
- 没有看到和 `更新显示血量` 类似的显式 `base.send_custom_event(...)`

这让链路存在一个疑点：

- 如果该引擎要求显式派发事件，那么当前点击商店逻辑可能没有真正发到服务端。
- 如果该引擎允许 `TriggerEvent + autoForward` 自动转发，那么还需要继续核对前后端定义是否一致。

#### 4. 同名事件的 `autoForward` 前后端不一致

同一个 `客户端点击商店` 事件：

- `ui/script/trigger_module_main_1.ts` 中 `autoForward = true`
- `script/trigger_module_main_1.ts` 中 `autoForward = false`

这个不一致本身就是风险点。即使引擎支持自动转发，也可能因为定义不一致导致行为异常。

### 最可能原因

按当前证据，优先级从高到低如下：

1. 客户端点击“重新打开商店”时，没有稳定地把事件传回服务端。
2. `客户端点击商店` 的 `autoForward` 前后端定义不一致，导致自动转发链路不稳定。
3. 商店关闭后，`smallcard_store` 面板仍然占位或拦截点击，但这一点需要库源码才能确认。

### 为什么会表现成“第一次可以，后面不行”

一个比较符合现象的解释是：

- 首次显示主要依赖 `MainPage` 模板中默认挂载的商店面板。
- 关闭以后，页面状态已经改变。
- 再次打开需要重新走“点击 -> 客户端事件 -> 服务端 `打开商店`”链路。
- 如果这条链路不完整，就会表现成“关了以后再也打不开”。

### 建议的修复方向

优先做这几步核对：

1. 明确 `客户端点击商店` 是否需要显式 `send_custom_event`，不要只靠“构造对象”。
2. 统一 `客户端点击商店` 在客户端和服务端的 `autoForward` 定义。
3. 检查关闭商店后，商店主面板是否只是隐藏，还是仍在拦截点击。
4. 如果你能找到编辑器安装目录中的 `smallcard_store` 源码，再补查关闭按钮到底改了哪些状态。

## 三、问题 2：商店内部配置不一样

### 现象理解

你说的“商店内部配置不一样”，结合当前代码，最可能是：

- 你看到的运行时商店内容，和你预期的“示例商店”不一样
- 或者页签、名字、商品列表和库里示例不一样

### 已确认的代码事实

#### 1. 当前运行时明确打开的是本地图 `游戏商店`

服务端 `script/trigger_module_main_1.ts` 里传入的是：

- `$$p_3b4f.loot_pool.游戏商店.root`

这已经说明运行时不是随便打开一个默认商店，而是明确绑定到当前地图自己的 `loot_pool`。

#### 2. 当前地图确实有自己的商店条目

`editor/table/entry_data/loot_pool/游戏商店/entry_data.ini` 中定义了：

- 商店名：`游戏商店`
- 页签名：`防御塔`
- 货币：`金钱`
- 商品：当前是 `item.塔1.root`

这是一套地图自定义商店数据。

#### 3. 本地化里同时存在“游戏商店”和“示例商店”

`ui/script/obj/localization/localization.lua` 同时包含：

- `$$p_3b4f.loot_pool.游戏商店.*`
- `$$smallcard_store.loot_pool.示例商店.*`

例如：

- `游戏商店` 的页签默认名是 `防御塔`
- `示例商店` 的页签默认名是 `装备 / 护符 / 消耗`

这说明仓库里同时存在两套概念：

- 一套是当前地图自定义商店
- 一套是 `smallcard_store` 库自带示例商店

### 最可能原因

最可能不是“配置被改坏了”，而是“你对比了两套不同的数据源”：

1. 当前地图真正打开的是 `游戏商店`
2. 你对照的可能是 `smallcard_store` 的 `示例商店`
3. 两者本来就不是同一套页签、商品和文案

### 建议的理解方式

以后排查“商店内容不一致”时，先分清三层：

1. 服务端到底传了哪个 `loot_pool_id`
2. `editor/table/entry_data/loot_pool/` 里该 `loot_pool` 的数据是什么
3. `localization.lua` 里该 `loot_pool` 对应的文案是什么

只要这三层对上，商店内容就基本可解释。

## 四、结论

### 对问题 1

当前最可疑的是“重新打开商店”的事件链路不完整，尤其是：

- 客户端点击逻辑未见显式派发
- `autoForward` 前后端不一致

### 对问题 2

当前最可信的解释是：你看到的是本地图自定义 `游戏商店`，不是库里的 `示例商店`，所以内部配置看起来不一样是正常现象。

## 五、后续排查建议

如果后面要继续修这个商店问题，推荐按下面顺序动手：

1. 先确认 `ui/src/gui/ui_response/单击商店按钮.lua` 的源逻辑是否应显式发事件。
2. 再统一 `客户端点击商店` 的前后端事件定义。
3. 最后再查 `smallcard_store` 库内部的关闭和重新显示逻辑。
