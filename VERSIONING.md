# 版本管理规则

## 分支规则

- `main`：稳定分支，只放可正常打开、可运行、可交付的版本。
- `feature/<功能名>`：新功能开发分支，例如 `feature/tower-upgrade`。
- `fix/<问题名>`：普通问题修复分支，例如 `fix/ui-cost-display`。
- `hotfix/<问题名>`：线上或已发布版本的紧急修复分支。
- `release/<版本号>`：发布前整理和测试分支，例如 `release/1.1.0`。

个人开发时可以直接在 `main` 提交小改动；多人协作或改动较大时，先建功能分支，确认能运行后再合并到 `main`。

## 版本号规则

使用语义化版本号：`主版本.次版本.修订号`，并用 Git 标签标记发布版本。

- `v1.0.0`：第一个稳定可玩版本。
- `v1.1.0`：新增玩法、系统、关卡、单位、装备等功能。
- `v1.1.1`：修复 bug、调整数值、优化表现，不改变主要玩法结构。
- `v2.0.0`：大规模重构、存档或玩法规则不兼容的版本。

发布时执行：

```bash
git tag v1.0.0
git push origin v1.0.0
```

## 提交信息规则

提交信息建议使用固定前缀，方便以后查历史：

- `feat:` 新功能
- `fix:` 修复问题
- `balance:` 数值调整
- `art:` 美术、特效、音效、模型、图标等资源调整
- `ui:` 界面和交互调整
- `docs:` 文档调整
- `refactor:` 重构，不改变实际玩法
- `chore:` 工程配置、忽略规则、整理文件

示例：

```bash
git commit -m "feat: add basic tower upgrade flow"
git commit -m "balance: adjust early wave monster health"
git commit -m "fix: correct shop refresh cost display"
```

## 提交流程

1. 开始改动前先拉取最新代码：`git pull --rebase origin main`。
2. 一次提交只做一类事情，避免把功能、修 bug、数值和资源整理混在一起。
3. 提交前检查状态：`git status`。
4. 确认项目能正常打开，关键玩法能跑通后再推送。
5. 拉取最新工程后，运行项目目录下的 `update_file.bat` 重置触发编辑器状态。

## 文件管理规则

- 需要版本管理：源码、地图配置、表格、场景、UI 配置、项目文档、正式资源。
- 不需要版本管理：本机编辑器配置、调试状态、触发器生成物、缓存文件、诊断文件。
- 不要提交包含个人路径、账号、设备信息或临时调试信息的文件。
- 大资源文件如果后续明显超过 GitHub 普通仓库使用范围，再考虑迁移到 Git LFS。

## 发布规则

每次准备发布时：

1. 从 `main` 创建 `release/<版本号>` 分支。
2. 只在发布分支做 bug 修复、数值微调和文档整理。
3. 测试通过后合并回 `main`。
4. 在 `main` 打标签，例如 `v1.1.0`。
5. 推送分支和标签到远端。
